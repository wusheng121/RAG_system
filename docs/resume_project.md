# RAG智能客服项目 - 简历描述与面试话术

## 简历项目描述（可直接粘贴）

### 版本A（1-2行，适合简历项目列表）
开发基于 FastAPI 的 RAG 智能客服与书店业务系统，打通搜索、购物车、结账、订单闭环；实现数据库直答 + 检索增强问答双通路，并加入模型降级和历史库兼容机制，提升问答稳定性与可用性。

### 版本B（3条要点，适合项目经历）
- 设计并实现 `FastAPI + SQLAlchemy + SQLite` 后端，完成图书检索、购物车、结账、订单查询等核心业务链路。
- 构建 RAG 客服流程（Query 扩展、BM25/向量混合检索、重排、LLM 生成），并对高频问题引入数据库直答策略，减少幻觉与接口失败率。
- 针对模型加载失败、旧数据库结构不一致等场景增加降级与兼容逻辑，补充 `smoke_test.py` 与 `pytest` 用例保障可回归。

### 版本C（STAR表达，适合网申长文本）
- **S（场景）**：需要把“练手RAG”升级为可展示的业务型项目，支持真实页面流程与客服问答。
- **T（任务）**：提升系统稳定性和可讲述性，避免高频问答与订单功能出现 500。

## 面试讲解话术（2-3分钟）

这个项目最初是一个 RAG 问答 demo，我后来把它改造成了一个“有业务场景”的智能客服系统。后端用 FastAPI，前端是模板页面，先把书店流程做完整：图书浏览、搜索、购物车、结账和订单查询。然后把客服接进业务数据，不只是查静态知识库，还能回答数据库里的实时问题，比如“有哪些书”“我购物车里有什么”“订单有哪些”。

技术上我做了两层回答策略：
1. 高频事实问题先走数据库直答，保证结果稳定且可解释；
2. 其他问题走 RAG（query 扩展 + 混合检索 + 重排 + 生成）。

我还重点处理了工程问题：模型不可用时自动降级，避免接口 500；历史订单表字段不一致时做兼容查询，保证旧数据也能用。最后用 `smoke_test.py` 和 `pytest` 做回归，确保每次改动后核心链路还能跑通。

## 高频面试追问参考

- **你为什么要做数据库直答？**
  因为库存、价格、订单这类事实问题不需要模型推理，直答可以降低幻觉、降低延迟，也更符合业务可信性。

- **RAG失败时怎么办？**
  在检索、重排、生成任一环节失败时，系统会退化到可用回答，不让前端直接报 500。

- **你做过哪些稳定性工作？**
  做了库存校验、异常兜底、历史库字段兼容、以及冒烟测试和接口测试。

- **如果继续优化你会做什么？**
  我会加后台知识库管理、埋点监控、Redis 缓存和更完整的权限体系。
